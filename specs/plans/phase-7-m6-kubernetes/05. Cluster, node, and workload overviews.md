# Phase 7 · Step 5 — Cluster, node, and workload overviews

**Milestone:** M6 · **Status:** ✅ Done (2026-05-27)

> **Landed.** Pure composition over the step-0 generic primitives. **Cluster
> Overview** (top "Overview" tree node) is a `PanelDashboard`: live CPU/Memory
> gauges + Pods/Nodes stat cards + CPU/mem time-series (metrics stream), the node
> list, and recent events. **Node detail** gains a live Metrics panel +
> scheduled-pods table; **workloads** (Deployment/StatefulSet/DaemonSet/
> ReplicaSet) gain an owned-pods table (matched by selector). Metrics read from
> **metrics-server** (`metrics.k8s.io`) and degrade gracefully when it is absent
> (`metricsAvailable:false`, lists still work). _Deferred:_ a richer namespace
> overview dashboard (currently generic detail + events) and the optional
> Prometheus adapter (config field present, adapter not yet wired).
**Depends on:** 7.0 (dashboard-as-view + metrics panel), 7.2 (resources) · **Spec refs:** [v2 §6.3](../../v2.md), [v2 §7.3](../../v2.md)

## Goal
The Lens-style at-a-glance surfaces (region 5 of the reference layout): a
**Cluster Overview**, **Node** detail, **Workload** overview, and **Namespace**
overview — each a multi-panel dashboard of gauges + time-series + lists, assembled
**only** from the generic `PanelDashboard` + metrics/stat panels delivered in
step 0. The k8s plugin contributes routes and manifest layout, **no frontend**.

> Cross-plugin reminder: this step *uses* generic primitives; it adds none. Any
> overview shape it needs that the renderer can't express is a step-0 gap, fixed
> generically there.

## Checklist
- [ ] **Cluster Overview** (connection landing or a top dashboard view): CPU /
      Memory / Pods **gauges** (used vs requests/limits/capacity), cluster CPU/mem
      **time-series**, node list with status, and a recent **Warnings/Events**
      table — composed via `PanelDashboard` child panels with `Span` hints.
- [ ] **Node detail**: capacity/allocatable, conditions, node metrics time-series,
      and the **pods scheduled on the node** (live table) in one dashboard view.
- [ ] **Workload overview** (Deployment/StatefulSet/DaemonSet): desired/ready/
      updated replicas, **pod status breakdown**, container images, and recent
      events — as the workload's "Overview" tab.
- [ ] **Namespace overview**: workload/pod counts, resource-quota usage, and
      events for the namespace.
- [ ] **Metrics source**: read from `metrics-server` (`metrics.k8s.io`) for
      live CPU/mem; optional Prometheus adapter behind a config toggle. Render
      through the generic metrics panel — the source is a k8s detail, the
      rendering is not.
- [ ] Empty/degraded states: overviews render cleanly when metrics-server is
      absent (gauges show "metrics unavailable", lists still work).

## Definition of done
Opening a cluster shows the Cluster Overview dashboard with **live** gauges and
time-series; a node and a Deployment each render a multi-panel overview; all panels
update via the watch/metrics path; nothing in the frontend is k8s-aware.

## Notes
This is where the cockpit "feels like Lens." It is deliberately last: it is the
biggest payoff but also pure composition over steps 0–4. If a gauge/threshold/
layout need appears that the metrics or dashboard panel can't express, add it to
step 0 generically (so Proxmox/Redis/brokers gain the same overview power).

## Phase 7 exit (M6)
Kubernetes is a **Lens / OpenLens-grade** cockpit — categorized nav, live kind
lists, resource details with multi-panel overviews, a resizable dock hosting
terminals/logs/YAML editors, create/edit/apply, and cluster/node/workload metrics
— working over **both** direct (kubeconfig) and **L7-agent** transport. Crucially,
the entire experience is **manifest-driven through generic, cross-plugin renderer
primitives** (step 0): the k8s package added zero frontend code, and the same dock
+ dashboards + metrics + nav primitives are available to every other plugin. The
full plugin model is proven across L4 and L7. Update
[`../README.md`](../README.md) Phase 7 to ✅.
