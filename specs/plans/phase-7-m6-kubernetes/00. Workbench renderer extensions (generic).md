# Phase 7 · Step 0 — Workbench renderer extensions (generic)

**Milestone:** M6 · **Status:** ☐ Not started
**Depends on:** Phase 1 (renderer), Phase 2 (contract) · **Spec refs:** [v2 §6](../../v2.md), [v2 §7](../../v2.md)

## Goal
Give the universal renderer the few primitives a **Lens / OpenLens-grade**
operations cockpit needs — a bottom **dock**, **multi-panel views**, **multiple
open workbench tabs**, **richer metrics**, and **list-opening navigation** — so
the Kubernetes plugin (steps 1–5) is pure manifest + routes with **zero
per-plugin frontend**.

> **Hard invariant — these are CROSS-PLUGIN primitives, not Kubernetes
> features.** k8s is only the forcing function. Every addition below must be
> generic, declared through the manifest projection, and rendered by the core.
> No `if plugin === "kubernetes"` anywhere, no k8s vocabulary in the contract.
> Each item lists non-k8s consumers, and its Definition of Done **must be proven
> on a non-k8s surface (a fixture or an existing plugin) first**. If a proposed
> shape can't be stated without naming a k8s concept, it's wrong — generalize it.

This step is **UI-first**: land and test every primitive on fixtures before any
k8s wiring exists (the project's phase rule).

## Checklist

### 1. Dockable bottom pane (the defining workbench element)
- [ ] Renderer: a resizable, collapsible **dock** region in the workspace shell,
      available in **all** layouts (`tabs`, `sidebar_tree`, `dashboard`), with its
      own closable/persistent tab strip; `KeepAlive` so terminals/streams survive
      navigation; respects `prefers-reduced-motion`; keyboard + ARIA.
- [ ] Contract: a generic **open-target** so a panel/stream can open **into the
      dock** instead of replacing the main view — e.g. `Action.Open: "view" |
      "dock" | "dialog"` (default `view`) and/or a `Tab`/panel placement hint.
      No new panel types required — `terminal`, `log_stream`, `code_editor`,
      `table`, `query_editor` simply render inside a dock tab.
- [ ] Cross-plugin consumers: SSH/Docker exec **terminal** in the dock while
      browsing files; database **query results** docked under the schema tree;
      Proxmox **task log** docked beside the VM list; any plugin's logs.
- [ ] DoD (non-k8s): from the SSH or Docker fixture, open a terminal **and** a
      log stream as two dock tabs, resize/collapse the dock, navigate the main
      view, and confirm both dock streams stay alive.

### 2. Multi-panel views (dashboard-as-a-view, not just connection-level)
- [ ] Contract: a `PanelDashboard` panel type whose config lists child panels
      (`{panel, source, config, span}`), usable as a **`Tab` panel** inside a
      `DetailView` or connection `Tabs`. Reuses the `LayoutDashboard` grid +
      `Tab.Span` already shipped, lifted to the panel level.
- [ ] Renderer: factor the existing `DashboardWorkspace` grid into a shared
      `DashboardPanel` so connection-level and view-level dashboards share code.
- [ ] Cross-plugin consumers: a database connection **Overview** tab (size +
      connections + activity); a broker **Overview** (rates + queues + consumers);
      a Proxmox **node** overview (metrics + VMs + tasks).
- [ ] DoD (non-k8s): a fixture resource detail renders an Overview tab that shows
      ≥ 3 child panels at once, span-aware, with no per-plugin code.

### 3. Workbench tabs (multiple open views)
- [ ] Workspace state: track **multiple open views** per connection (resource
      details, kind lists, overviews) as a closable/pinnable top tab strip,
      instead of a single selected ref. Reopen restores; close/X and middle-click.
- [ ] Contract impact: **none** — the open set is user/session state; the
      projection is unchanged. Pure renderer/store work.
- [ ] Cross-plugin: works for every `sidebar_tree` plugin (open two DB tables
      side-by-side as tabs; keep a Docker container detail open while browsing).
- [ ] DoD (non-k8s): open three resource views from a fixture tree, switch and
      close them; closing the active tab falls back to a sensible neighbor.

### 4. Metrics / stat panel richness
- [ ] Contract: a `MetricsConfig` (series, unit, `kind: timeseries | gauge |
      stat`, thresholds, optional `requests`/`limits`/`capacity` overlays) and/or
      a lightweight `stat` panel for KPI cards. Plugin-agnostic field names.
- [ ] Renderer: time-series chart + gauge/stat cards (reuse an existing charting
      lib via context7-verified API; skeleton/empty/error states).
- [ ] Cross-plugin consumers: Proxmox node/VM metrics, Redis memory/ops, broker
      throughput, DB connection counts — all render the same panel.
- [ ] DoD (non-k8s): a fixture renders a gauge card and a multi-series time-series
      from a metrics route.

### 5. List-opening navigation nodes
- [ ] Contract: let a tree node target a **resource-kind list view** (or an
      arbitrary panel/view), not only a single-resource `Ref` detail — e.g.
      `TreeNode.ResourceKind` / a `view` target — and allow **nested** tree
      groups (category → kind). Stays generic (a "kind" is any resource type).
- [ ] Renderer: clicking such a node opens that kind's `list` (live `table`) in
      the main area / a workbench tab, exactly like a group-click does today.
- [ ] Cross-plugin: a database "Tables" category that expands to schemas, each
      opening a table-list view; a storage plugin's nested buckets.
- [ ] DoD (non-k8s): a fixture tree nests a category whose leaf opens a kind list
      (table), distinct from leaves that open a single-resource detail.

### 6. Create-from-content (generic "apply a document")
- [ ] Contract: compose from existing parts — a connection/resource **action**
      that opens a `code_editor` **in the dock**, seeded from optional
      **templates** (`templatesRouteId` on the editor config), saved via a route
      with **dry-run + diff**. No new panel; small config + the dock from item 1.
- [ ] Cross-plugin consumers: "Run a saved snippet" (SSH), "New document"
      (Mongo), "Apply config" — any plugin that authors text and submits it.
- [ ] DoD (non-k8s): a fixture opens a templated editor in the dock and submits it
      through a save route (dry-run path exercised).

## Definition of done
On **fixtures only** (no k8s plugin yet): the workspace shows a resizable dock
hosting a terminal + log + editor as tabs; a resource detail renders a multi-panel
Overview tab; three views are open as closable workbench tabs; a nav leaf opens a
kind list; a metrics route renders gauge + time-series. Go contract types + TS
projection mirror + validator updated; `make fmt && make lint && make test` green;
**every new field documented in [`../../v2.md`](../../v2.md) as generic, with a
non-k8s example.**

## Notes
This is the only step that touches the renderer/contract. Steps 1–5 must add
**no** frontend code — if they need something the renderer can't express, it
comes back here as a generic primitive, never a k8s special case.
