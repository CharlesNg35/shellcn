# Phase 7 · Step 3 — Pod logs, exec, and port-forward

**Milestone:** M6 · **Status:** ☐ Not started
**Depends on:** 7.0 (dock), 7.2, 3.3, 4.3 · **Spec refs:** [v2 §8.1](../../v2.md), [v2 §12](../../v2.md)

## Goal
Pod-level streaming over the L7 transport (these use SPDY/WebSocket upgrades, not
a raw socket), surfaced through the Lens-style **bottom dock** (regions 6–7) via
the generic dock primitive from step 0 — declared in the manifest, no k8s UI.

## Checklist
- [ ] Pod logs WS (container select, follow, previous) → `log_stream` panel, opened **in the dock** (manifest action with `Open: "dock"`)
- [ ] Pod exec WS (TTY, resize) → `terminal` panel, opened **in the dock**
- [ ] Multiple concurrent dock tabs (logs from several pods + an exec shell at once), each a manifest-declared action target — reusing the generic dock, not a k8s widget
- [ ] Port-forward (channel) → tunnel action; surface the active forward
- [ ] All working via **both** direct (kubeconfig) and **agent** (L7) transport
- [ ] Backpressure + reconnect

## Definition of done
Logs, exec, and port-forward all work against a pod via both direct and agent
transport, opening as dock tabs that persist while the user navigates — driven
entirely by manifest action targets, with no k8s-specific frontend.

## Notes
Confirms the L7 transport handles client-go's stream upgrades — the hardest agent
case. The dock itself is generic (step 0): SSH/Docker exec and DB query results use
the same mechanism.
