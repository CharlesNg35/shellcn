# Phase 7 · Step 2 — Workloads and core resource trees

**Milestone:** M6 · **Status:** ✅ Done (2026-05-27)

> **Landed.** A single **kind catalog** (`kinds.go`) drives the Lens-style menu
> (Nodes / Workloads / Config / Network / Storage / Namespaces / Events / Access
> Control / Custom Resources), the manifest `ResourceType`s, and one set of
> **generic, `{kind}`-parameterized** routes (list/get/watch/delete/scale/
> restart/cordon) that serve every built-in kind and every runtime-discovered
> CRD — adding a kind is one catalog entry, zero new routes/frontend. ~25
> built-in kinds with curated columns; CRDs discovered via the dynamic client and
> rendered through a generic `customresource` type; live `ResourceEvent` watch;
> namespace scoping; Secrets redacted (no data values). Code is split across
> small files (kinds, mappers, kindrows, generic, watch, tree, crd, resources).
**Depends on:** 7.0 (nav primitives), 7.1 · **Spec refs:** [v2 §6.3](../../v2.md), [v2 §7.3](../../v2.md)

## Goal
The manifest's resource trees and types, with informer-backed live updates,
rendered with Lens-style categorized navigation (region 3 of the reference
layout) using only the generic tree primitives from step 0. **Everything is
manifest-driven**: the plugin emits a projection + routes; the core renders it
with no k8s-specific frontend.

## Checklist
- [ ] **Nested categories** (step-0 list-opening nodes): Workloads / Network /
      Storage / Config / Access Control (RBAC) / Namespaces / Events / Custom
      Resources — each declared in the manifest tree, expanding to its **kinds**;
      a kind leaf opens that kind's **live list view** (not a single-resource
      detail).
- [ ] Resource types: Deployments, Pods, StatefulSets, DaemonSets, ReplicaSets, Jobs/CronJobs, Services, Ingress, PVC/PV, ConfigMaps, **Secrets (redacted)**
- [ ] **CRDs**: discover custom resource kinds at runtime and list them under
      Custom Resources via route-driven tree children (manifest declares the
      shape; instances/kinds come from routes — no hardcoding).
- [ ] Columns + actions per type (scale/restart/delete with risk tags), all declared in the manifest
- [ ] Watches via informers → `ResourceEvent` (live table/tree patching)
- [ ] Namespace scoping/selector
- [ ] Per-resource **Overview** tab is a manifest-declared `PanelDashboard` (primitive from 7.0; content in 7.5); YAML/Events/Logs tabs follow in 7.3–7.4

## Definition of done
The cluster renders as categorized nav → live kind lists → resource details and
updates live as workloads change state — entirely from the manifest projection
through the generic renderer, with no k8s-specific frontend.

## Notes
Largest surface in the project — it leans entirely on the proven generic renderer
+ watch path, plus the step-0 nav/list primitives. Nothing here is bespoke UI: a
"kind" is a generic resource type and a "category" is a nested tree group any
plugin can declare.
