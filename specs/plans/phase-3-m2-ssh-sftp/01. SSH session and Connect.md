# Phase 3 · Step 1 — SSH session and Connect

**Milestone:** M2 · **Status:** ✅ Done (2026-05-25)
**Depends on:** Phase 2 · **Spec refs:** [v2 §8.1](../../v2.md), [v2 §9.6](../../v2.md), [v2 §13](../../v2.md)

## Goal
The reference plugin's session: one authenticated `ssh.Client` multiplexing
terminal + SFTP + tunnels, with strict host-key verification.

## Checklist
- [x] `Connect` dials via `cfg.Net.DialContext` (works direct now; agent later)
- [x] Auth methods: password, private key, agent
- [x] **Strict host-key verification** — "ignore host key" is never the default
- [x] First-connection / changed-key flow: `known_hosts` import + **TOFU approval** surfaced as a generic `VerificationRequired{ kind: "host_key" }` challenge (§9.6); on accept, append to `known_hosts`
- [x] `sshSession{ client, mu, sftp }`; SFTP opened lazily under the mutex
- [x] `HealthCheck` (keepalive) + `Close`

## Definition of done
Connect to a test SSH server; an unknown key raises the verification challenge (accept → trusted, persisted); a *changed* key is rejected pending approval; SFTP sub-client opens lazily without a second TCP connection.

## Tests (gate)
- [x] Unknown key → `VerificationRequired`; accept persists to `known_hosts`; reject aborts
- [x] Changed key is **not** silently trusted
- [x] `-race`: concurrent Terminal + SFTP open shares one client without a data race

## Notes
This validates the §8 concurrency model (stateless plugin, state in Session, mutex-guarded sub-client) and the generic verification-challenge mechanism (§9.6) — no SSH-specific frontend code.
