# Phase 3 · Step 1 — SSH session and Connect

**Milestone:** M2 · **Status:** ☐ Not started
**Depends on:** Phase 2 · **Spec refs:** [v2 §8.1](../../v2.md), [v2 §9.6](../../v2.md), [v2 §13](../../v2.md)

## Goal
The reference plugin's session: one authenticated `ssh.Client` multiplexing
terminal + SFTP + tunnels, with strict host-key verification.

## Checklist
- [ ] `Connect` dials via `cfg.Net.DialContext` (works direct now; agent later)
- [ ] Auth methods: password, private key, agent
- [ ] **Strict host-key verification** — "ignore host key" is never the default
- [ ] First-connection / changed-key flow: `known_hosts` import + **TOFU approval** surfaced as a generic `VerificationRequired{ kind: "host_key" }` challenge (§9.6); on accept, append to `known_hosts`
- [ ] `sshSession{ client, mu, sftp }`; SFTP opened lazily under the mutex
- [ ] `HealthCheck` (keepalive) + `Close`

## Definition of done
Connect to a test SSH server; an unknown key raises the verification challenge (accept → trusted, persisted); a *changed* key is rejected pending approval; SFTP sub-client opens lazily without a second TCP connection.

## Tests (gate)
- [ ] Unknown key → `VerificationRequired`; accept persists to `known_hosts`; reject aborts
- [ ] Changed key is **not** silently trusted
- [ ] `-race`: concurrent Terminal + SFTP open shares one client without a data race

## Notes
This validates the §8 concurrency model (stateless plugin, state in Session, mutex-guarded sub-client) and the generic verification-challenge mechanism (§9.6) — no SSH-specific frontend code.
