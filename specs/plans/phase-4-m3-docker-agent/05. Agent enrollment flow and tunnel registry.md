# Phase 4 · Step 5 — Harden enrollment/tunnel registry with Docker agent mode

**Milestone:** M3 · **Status:** ✅ Done (2026-05-26)
**Depends on:** 4.2, 4.4 · **Spec refs:** [v2 §8.4](../../v2.md), [v2 §9.5](../../v2.md)

## Goal

Harden the existing enrollment lifecycle and core-side tunnel registry with the
first real agent-mode plugin. Phase 2 introduced the generic endpoint/token/tunnel
path; this step finishes production Docker behavior and missing artifact variants.

## Checklist

- [x] Existing `POST /api/connections/{connectionID}/agent/enrollments` proven with Docker agent connections
- [x] Artifacts: inline `command` (token via `SHELLCN_ENROLL_TOKEN` env, never in a path); defer `k8s-manifest` URL artifact to Kubernetes/L7 unless Docker needs a downloadable artifact
- [x] Artifact fetch endpoint/ticket flow implemented only for artifact kinds that cannot be safely embedded inline
- [x] Agent connect path proven under token replay, expiry, reconnect, and disconnect cases
- [x] Tunnel registry status is accurate for Docker workspace gating: `pending → online → offline`
- [x] Audit enrollment create, artifact fetch where present, agent connect/disconnect/re-enroll
- [x] `EnrollPanel` reflects live Docker agent status without refresh

## Definition of done

Running the generated `docker run ghcr.io/charlesng35/shellcn-agent …` command (token via env var) flips the connection to **online**, and the `EnrollPanel` updates without refresh.

## Notes

Credential is scoped to one connection + one `ProxyTarget` — it cannot be repurposed (§8.4 security).
This step depends on 4.2 because the enrollment artifacts and `ProxyTarget` come
from the plugin-declared `AgentProfile`, not from hardcoded Docker-specific UI.
