# Phase 4 · Step 5 — Agent enrollment flow and tunnel registry

**Milestone:** M3 · **Status:** ☐ Not started
**Depends on:** 4.2, 4.4 · **Spec refs:** [v2 §8.4](../../v2.md), [v2 §9.5](../../v2.md)

## Goal

The enrollment lifecycle: install artifact → token → tunnel → online, plus the
core-side tunnel registry.

## Checklist

- [ ] `POST /api/connections/{connectionID}/agent/enrollments` → mint short-lived, single-use token; return `{ enrollmentId, expiresAt, artifacts[] }`
- [ ] Artifacts: inline `command` (token via `SHELLCN_ENROLL_TOKEN` env, never in a path) + fetch `url` for `k8s-manifest`
- [ ] `GET /api/connections/{connectionID}/agent/enrollments/{enrollmentId}/artifacts/{kind}` guarded by a single-use signed `?ticket=` (§7.4) — for kubectl/curl
- [ ] Agent connect: global `wss://host/api/agent/connect`; token presented **in the handshake**, not the URL → issue long-lived, connection-scoped credential
- [ ] Tunnel registry: connection ↔ live tunnel; status `pending → online`; reconnect/credential rotation
- [ ] Audit enrollment create, artifact fetch, agent connect/disconnect/re-enroll
- [ ] `EnrollPanel` reflects live status (1.6)

## Definition of done

Running the generated `docker run shellcn-proxy …` command (token via env var) flips the connection to **online**, and the `EnrollPanel` updates without refresh.

## Notes

Credential is scoped to one connection + one `ProxyTarget` — it cannot be repurposed (§8.4 security).
This step depends on 4.2 because the enrollment artifacts and `ProxyTarget` come
from the plugin-declared `AgentProfile`, not from hardcoded Docker-specific UI.
