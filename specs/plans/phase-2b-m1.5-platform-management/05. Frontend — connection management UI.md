# Phase 2b · Step 5 — Frontend connection management UI

**Milestone:** M1.5 · **Status:** ✅ Done
**Depends on:** 2b.1, 2b.4 · **Spec refs:** [v2 §12.2](../../v2.md), [v2 §6](../../v2.md)

## Goal
Create / edit / delete connections from the UI — **manifest-driven**, reusing the
generic `SchemaForm`, so a new plugin needs zero management-UI code.

## Checklist
- [x] "Add connection": plugin picker (from `GET /api/plugins` summaries) → fetch projection → render its `config` `Schema` with the existing `SchemaForm` (incl. `VisibleWhen`, validators, `credential_ref` selector)
- [x] Shared-manager edit redacts inaccessible credential refs, shows a keep-or-replace affordance, and sends `preserveCredentials` instead of hidden credential IDs
- [x] Transport selector (`direct`/`agent`) limited to the plugin's `SupportedTransports`; agent connections rely on the existing `PanelEnroll` once created
- [x] Submit → `POST /api/connections`; on success route into the workspace
- [x] Edit: prefill from the connection-detail read; `Secret` fields show `set`/`not set` + "replace" (never the value)
- [x] Delete with confirm (PrimeVue `Dialog`/confirm); toast feedback
- [x] Empty/loading/error states; optimistic list refresh; accessible + keyboard-operable

## Definition of done
A user adds a connection by picking a plugin and filling its schema-rendered form (inline secret or referenced credential), edits it (without ever seeing stored secrets), and deletes it — all generic, no per-plugin code.

## Tests (gate)
- [x] Component: form renders from a fixture projection; required/validator errors block submit; secret field shows `set`/replace on edit
- [x] e2e: add → appears in list/opens; edit → persists; delete → removed

## Notes
Reuse `SchemaForm` + `CredentialSelect` (already built). Do not special-case any plugin (AGENTS.md invariant).
