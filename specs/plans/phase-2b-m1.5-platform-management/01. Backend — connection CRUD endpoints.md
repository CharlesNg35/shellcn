# Phase 2b · Step 1 — Backend connection CRUD endpoints

**Milestone:** M1.5 · **Status:** ✅ Done
**Depends on:** 2.3, 2.4, 2.6, 2.8 · **Spec refs:** [v2 §12.2](../../project.md), [v2 §6.1](../../project.md), [v2 §9.3](../../project.md)

## Goal
Add the platform control-plane endpoints to create, edit, and delete connections.
Today only list/read exists.

## Checklist
- [x] `POST /api/connections` — create; `PUT /api/connections/{id}` — update; `DELETE /api/connections/{id}`
- [x] Validate the submitted config against the plugin's `config` `Schema` (reuse `RequestContext.ValidateSchema` / the schema validators), incl. `VisibleWhen` + required + validators
- [x] Split secret vs non-secret fields by `Field.Secret`; encrypt secrets via the service layer before the store (store sees ciphertext only)
- [x] Write-only secret semantics on update: a field omitted/blank keeps the stored value; an explicit new value replaces it; never return secret values
- [x] `credential_ref` fields store only the credential ID; create/edit validates replacements against the acting user, edit can preserve an unreadable existing credential, and shared-connection runtime resolves the bound credential through the connection owner
- [x] Transport (`direct`/`agent`) accepted + validated against the plugin's `SupportedTransports`
- [x] AuthZ: creator becomes owner; update/delete require owner/manage/admin; audited
- [x] A connection-detail read (`GET /api/connections/{id}`) returning non-secret config + per-field `set`/`not set` (no secret values) for the edit form
- [x] Tighten grant semantics for control-plane access: `use` can open/use a connection; `manage` is required to edit/share/delete unless owner/admin

## Definition of done
A user can create, edit, and delete a connection over the API; secrets are encrypted at rest and never returned; config is schema-validated; every mutation is audited.

## Tests (gate)
- [x] Create→get→update→delete round-trip; secret set on create, "replace" on update, untouched when omitted
- [x] Invalid config (schema/validator failure) → 400; unknown plugin → 400/404
- [x] AuthZ: non-owner without manage grant → 403; secret never appears in any response body

## Notes
The connection form on the FE renders from the plugin projection's `config` (§12.2) — this step is the API it submits to.
