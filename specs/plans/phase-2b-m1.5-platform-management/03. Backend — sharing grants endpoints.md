# Phase 2b · Step 3 — Backend sharing grants endpoints

**Milestone:** M1.5 · **Status:** ✅ Done
**Depends on:** 2.3, 2.6 · **Spec refs:** [v2 §9.2](../../v2.md), [v2 §12.2](../../v2.md)

## Goal
Let an owner share a connection or credential with another user. `GrantStore` /
`CredentialGrantStore` exist; this exposes the API + the policy hookup.

## Checklist
- [x] Connection grants: `POST /api/connections/{id}/grants` (subject + access `use`/`manage`), `GET` list, `DELETE /api/connections/{id}/grants/{grantId}`
- [x] Credential grants: `POST /api/credentials/{id}/grants` (access `use`), `GET` list, `DELETE`
- [x] Minimal subject lookup for sharing: `GET /api/users?query=` returns non-secret user summaries; full user management stays M-Admin
- [x] Only the owner (or admin) may grant/revoke; audited
- [x] Connection `use` permits opening/using a connection; connection `manage` permits edit/share/delete; credential `use` permits connect-time secret resolution but never secret readback
- [x] List returns subjects + access (resolve usernames for display; never secrets)
- [x] Granted access is honored by the route wrapper's authz, control-plane CRUD authz, and `CredentialService` use-checks

## Definition of done
An owner can grant/revoke `use`/`manage` on a connection and `use` on a credential; the grantee gains exactly that access (verified through the existing authz path); revocation takes effect immediately.

## Tests (gate)
- [x] Grant `use` → grantee can open the connection but not edit/delete; `manage` → can edit/share; revoke → access gone
- [x] Credential `use` grant → grantee can connect through it, never read the value; non-owner cannot grant → 403

## Notes
The authz gate (2.6/2.8) already consults grants for connection use. This step also tightens control-plane semantics so only `manage` grants can edit/share/delete.
