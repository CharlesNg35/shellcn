# Phase 2 · Step 3 — ent schema and store repositories

**Milestone:** M1 · **Status:** ☐ Not started
**Depends on:** 2.1 · **Spec refs:** [v2 §11.1](../../v2.md), [v2 §14.1](../../v2.md)

## Goal
The cross-database control-plane store via `ent`, hidden behind repository
interfaces so the ORM never leaks.

## Checklist
- [ ] ent schema: `User`, `Role`, `Grant`, `Connection` (+ encrypted secret columns), `AuditEntry`, `Snippet`, `AgentEnrollment`, `Preference`
- [ ] `go generate` the ent client
- [ ] Repository interfaces in `internal/store`: `UserStore`, `ConnectionStore`, `AuditStore`, `GrantStore`, …
- [ ] Driver wiring: `modernc.org/sqlite` (default) + `pgx` (Postgres) + `go-sql-driver/mysql`
- [ ] In-memory fakes implementing the interfaces (for unit tests)
- [ ] Startup auto-migration, GORM-style: create missing tables/indexes and apply additive schema changes only
- [ ] Guardrail: destructive schema/data changes are never automatic
- [ ] **Nothing outside `internal/store` imports `ent`**

## Definition of done
A fresh **SQLite** database boots and creates its schema automatically; CRUD works on SQLite (the M1 gate); the same store suite is parameterized to also run on Postgres + MySQL; no `ent` import leaks out of `store`.

## Tests (gate)
- [ ] Store suite green on **SQLite** (per-PR gate — fast)
- [ ] Same suite runs on **Postgres + MySQL** nightly / as an M1 hardening task (not blocking early PRs — see [`../TESTING.md`](../TESTING.md))
- [ ] Lint/architecture test: no package outside `internal/store` imports `ent`

## Notes
Secrets are encrypted in the service layer (2.4) before reaching the store — the store only sees ciphertext. SQLite stays the zero-config default; cross-DB is opt-in (v2 §11.1). Reviewed ent/Atlas migrations remain a later hardening option once the schema stabilizes.
