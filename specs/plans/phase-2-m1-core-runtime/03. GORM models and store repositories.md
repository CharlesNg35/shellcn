# Phase 2 · Step 3 — GORM models and store repositories

**Milestone:** M1 · **Status:** ☐ Not started
**Depends on:** 2.1 · **Spec refs:** [v2 §11.1](../../v2.md), [v2 §14.1](../../v2.md)

## Goal
The cross-database control-plane store via `gorm`, hidden behind repository
interfaces so the ORM never leaks.

## Checklist
- [ ] GORM models: `User`, `Role`, `Grant`, `Connection` (+ encrypted inline secret columns), `Credential` (+ encrypted secret material), `CredentialGrant`, `AuditEntry`, `Snippet`, `AgentEnrollment`, `Preference`
- [ ] Repository interfaces in `internal/store`: `UserStore`, `ConnectionStore`, `CredentialStore`, `AuditStore`, `GrantStore`, …
- [ ] Driver wiring: `glebarez/sqlite` (pure-Go default) + `gorm.io/driver/postgres` (pgx) + `gorm.io/driver/mysql` — never the cgo `gorm.io/driver/sqlite`
- [ ] In-memory fakes implementing the interfaces (for unit tests)
- [ ] Startup auto-migration via `AutoMigrate`: create missing tables/indexes and apply additive schema changes only
- [ ] Guardrail: destructive schema/data changes are never automatic
- [ ] Credential references: connections can point at reusable credentials without duplicating secret material
- [ ] Credential grants: `use` access permits connect-time resolution but never readback of secret values
- [ ] **Nothing outside `internal/store` imports `gorm`**

## Definition of done
A fresh **SQLite** database boots and creates its schema automatically via `AutoMigrate`; CRUD works for users, connections, credentials, and grants on SQLite (the M1 gate); the same store suite is parameterized to also run on Postgres + MySQL; no `gorm` import leaks out of `store`.

## Tests (gate)
- [ ] Store suite green on **SQLite** (per-PR gate — fast), including connection → credential reference CRUD
- [ ] Same suite runs on **Postgres + MySQL** nightly / as an M1 hardening task (not blocking early PRs — see [`../TESTING.md`](../TESTING.md))
- [ ] Lint/architecture test: no package outside `internal/store` imports `gorm`

## Notes
Secrets are encrypted in the service layer (2.4) before reaching the store — the store only sees ciphertext and stable credential references. SQLite stays the zero-config default; cross-DB is opt-in (v2 §11.1). Versioned migrations remain a later hardening option once the schema stabilizes.
