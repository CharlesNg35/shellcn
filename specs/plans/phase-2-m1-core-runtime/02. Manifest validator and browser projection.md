# Phase 2 · Step 2 — Manifest validator and browser projection

**Milestone:** M1 · **Status:** ✅ Done (2026-05-25)
**Depends on:** 2.1 · **Spec refs:** [v2 §5.1](../../v2.md), [v2 §5.2](../../v2.md)

## Goal
Validate every plugin manifest at registration, and derive the browser projection
(render-only, no executable internals).

## Checklist
- [x] Validate: `APIVersion` supported; unique IDs; every `RouteID` referenced by a Tab/Action/Stream exists
- [x] Validate: `AgentProfile` present iff `TransportAgent` declared
- [x] Projection builder: strip handler funcs, raw paths, permission keys, audit-event names
- [x] Projection keeps: identity, schema, layout, tabs/tree, columns, actions (`risk` + `requiresConfirm` + params), stream kinds, panel configs, `DataSource` (`RouteID` + params)
- [x] Golden projection covers the M0 panel configs: file browser operation routes, editable form submit, code-editor save, query cancel, and multipart-capable schema/action inputs.
- [x] Serve `GET /api/plugins` and `GET /api/plugins/{name}`
- [x] Unit tests for valid + each invalid case

## Definition of done
Invalid manifests are rejected with actionable errors; the projection output matches the Phase 1 fixtures' shape (golden test).

## Notes
This is the seam that lets the M0 UI run unchanged against the real server.
