# Phase 2 · Step 4 — Secret vault

**Milestone:** M1 · **Status:** ✅ Done (2026-05-25)
**Depends on:** 2.3 · **Spec refs:** [v2 §9.3](../../v2.md)

## Goal
Encrypt inline connection secrets and reusable credentials at rest behind a
`SecretStore` interface (OpenBao drop-in later).

## Checklist
- [ ] `SecretStore` interface (`internal/secrets`)
- [ ] Local AES-256-GCM vault; master key from env/file (KMS-ready)
- [ ] Per-record data key wrapped by the master key
- [ ] Encrypt `Field.Secret == true` inline values and reusable credential secret material; persist only ciphertext
- [ ] **Write-only API**: never return secret values; expose "set / not set" for inline secrets and credentials
- [ ] Redaction helpers used by logs + audit
- [ ] Connect-time credential resolver: authorized `credential_ref` IDs decrypt into plugin config without exposing the value to the client
- [ ] Credential rotation path documented: update once, all referencing connections use the new value
- [ ] Master-key rotation path documented

## Definition of done
Inline secrets and reusable credentials persist as ciphertext, never leave the API, never appear in logs/audit; encrypt→decrypt round-trip and credential-reference resolution tests pass.

## Notes
Encryption happens in the service layer, independent of the DB engine (2.3). A user with credential `use` permission can connect through that credential but cannot read, export, or copy its secret material.
