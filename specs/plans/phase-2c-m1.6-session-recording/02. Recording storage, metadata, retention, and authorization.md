# Phase 2c · Step 2 — Recording storage, metadata, retention, and authorization

**Milestone:** M1.6 · **Status:** ☐ Not started
**Depends on:** 2.3, 2.6, 2.9, 2c.1 · **Spec refs:** [v2 §9.2](../../v2.md), [v2 §9.5](../../v2.md), [v2 §11.1](../../v2.md)

## Goal
Persist recording metadata in the control-plane store and recording blobs in a
replaceable storage backend. Define who can see recordings.

## Checklist
- [ ] Add `Recording` model: id, user, connection, protocol, route/stream id, class, format, status, started/ended, size, checksum, storage key, retention/expires-at, error
- [ ] Add `RecordingStore` interface + GORM/in-memory implementations; include it in `store.Store`
- [ ] Add blob storage interface: local filesystem default, with S3/object storage shape kept behind an interface
- [ ] Authorization rules: admin sees all; normal users see their own recordings; owner/manage grant can see recordings for a connection if policy allows
- [ ] Append audit rows for recording start/stop/finalize/read/delete
- [ ] Retention policy: default keep, optional per-connection days; cleanup job deletes expired blobs + marks metadata
- [ ] Never store secrets in metadata; params are redacted like audit params

## Definition of done
Recording metadata and blobs persist separately; listing and reading recordings obey role/grant rules; expired recordings are cleaned up safely.

## Tests (gate)
- [ ] Store round-trip for metadata on SQLite + memory fake
- [ ] Blob storage write/read/delete tests with temp dir
- [ ] AuthZ tests: admin all, normal user own only, manage grant behavior, stranger denied
- [ ] Retention cleanup marks/deletes only expired recordings

## Notes
Recordings are larger and more sensitive than audit rows. They need explicit read authorization, retention, and deletion behavior from the start.
