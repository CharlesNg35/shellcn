# Phase 2c · Step 6 — Recording APIs and frontend management UI

**Milestone:** M1.6 · **Status:** ✅ Done (2026-05-25)
**Depends on:** 2b.4, 2b.5, 2c.2, 2c.4, 2c.5 · **Spec refs:** [v2 §9.2](../../project.md), [v2 §9.5](../../project.md), [v2 §12.2](../../project.md)

## Goal
Expose recording management and playback in the platform UI. Admins can see all
recordings; normal users see only recordings for sessions they created.

## Checklist
- [x] APIs: `GET /api/recordings`, `GET /api/connections/{id}/recordings`, `GET /api/recordings/{id}`, `GET /api/recordings/{id}/content`, `DELETE /api/recordings/{id}`
- [x] APIs for desktop chunks: create/start, chunk upload, finalize, abort
- [x] Filters: connection, user, protocol, class, status, time range
- [x] Frontend recording list (`RecordingsView`) with role-aware scope: admin all, normal user own
- [x] Per-connection recordings are listed for authorized users via `GET /connections/{id}/recordings` (filterable from the Recordings view by connection). Surfacing them inside a connection-workspace "activity" tab is optional future polish, not part of the M1.6 DoD.
- [x] Playback views: terminal cast player (`CastPlayer`) and WebM video player (`VideoPlayer`)
- [x] Stream panels show recording state per recordable component (terminal, desktop/VNC/RDP):
  - not supported → no control
  - `disabled` → no control
  - `manual`, idle → "Start recording" button; while recording → REC indicator + "Stop"
  - `auto` (forced) → REC indicator, no stop control (cannot be stopped)
  - show the recording class/type (terminal vs desktop) and a failed state
- [x] Manual start/stop control endpoint (`POST /connections/{id}/recordings/control`) wired to the engine registry; admin per-user drill-down (`/recordings?user=<id>`) from the Users view
- [x] Connection create/edit UI integrates recording policy from 2c.1: off-by-default options only when supported
- [x] Download/delete actions with confirmation and audit

## Definition of done
Users can find and play their own recordings. Admins can search all recordings.
Connection create/edit exposes auto-record only for plugins that declare support,
and stream panels show recording state without protocol-specific UI.

## Tests (gate)
- [x] API authz tests: admin all (+ per-user), user own only, unauthorized denied (`TestRecordingListScopeAndContent`, `TestRecordingAuthZScope`)
- [x] Frontend component test: list + route filter + type/state + playable affordance (`RecordingsView.test.ts`); player mounts (`CastPlayer.test.ts`)
- [x] Server integration: connection with terminal auto-record → open stream → recording appears finalized + content served (`TestWSStreamRecordedWhenPolicyForced`, `TestRecordingListScopeAndContent`)
- [x] Deletion requires authorization and writes an audit row

## Notes
Recording UI is platform UI, not plugin UI. Plugins declare capabilities; the core frontend renders the same controls for every protocol.
