# Phase 2c · Step 4 — Terminal asciicast recorder and playback

**Milestone:** M1.6 · **Status:** ☐ Not started
**Depends on:** 2c.2, 2c.3 · **Spec refs:** [v2 §9.5](../../v2.md), [v2 §12](../../v2.md)

## Goal
Implement the generic terminal recorder using asciicast v2 so SSH, Docker exec,
Kubernetes exec, telnet, and serial can share one recording/playback path.

## Checklist
- [ ] Write asciicast v2 header with width/height/timestamp/title/env allowlist
- [ ] Record upstream terminal output as `o` events
- [ ] Record resize as `r` events from the terminal resize control channel
- [ ] Browser input recording (`i` events) is optional and disabled by default; if enabled, mark recording sensitivity clearly
- [ ] Incremental write format; finalized recording remains readable if the session ends unexpectedly
- [ ] Frontend playback uses `asciinema-player` or a maintained equivalent verified before implementation
- [ ] Download endpoint serves `application/x-asciicast`

## Definition of done
The core can produce a valid `.cast` file from a fake terminal stream, and the frontend can play it back without knowing whether it came from SSH, Docker exec, Kubernetes exec, telnet, or serial.

## Tests (gate)
- [ ] Unit: asciicast header/events validate against known fixtures
- [ ] Unit: resize events are encoded as `{cols}x{rows}`
- [ ] Unit: input events are absent unless explicitly enabled
- [ ] Frontend component test: playback view loads a sample cast
- [ ] Golden sample recording checked into testdata

## Notes
Use asciicast as the primary terminal format because it is compact, incremental, and replayable in browser. `xterm.js` serialization is useful for restoring terminal state, not as the storage format for audit recordings.
