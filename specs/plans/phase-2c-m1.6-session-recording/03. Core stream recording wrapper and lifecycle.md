# Phase 2c · Step 3 — Core stream recording wrapper and lifecycle

**Milestone:** M1.6 · **Status:** ☐ Not started
**Depends on:** 2.8, 2.9, 2c.1, 2c.2 · **Spec refs:** [v2 §8.1](../../v2.md), [v2 §9.5](../../v2.md)

## Goal
Record stream traffic at the core wrapper boundary without making SSH, VNC, or
RDP special cases. The wrapper decides whether recording is active from plugin
capability + connection policy + user permissions.

## Checklist
- [ ] Add `Recorder` / `RecordingSink` interfaces under `internal/recording`
- [ ] Wrap `ClientStream` reads/writes so recorders can observe browser→upstream and upstream→browser traffic with timestamps
- [ ] Record lifecycle: pending → active → finalized / failed / discarded
- [ ] If a connection policy requires recording and the recorder cannot start, deny the stream before opening the upstream session
- [ ] Preserve backpressure: recording writes must not block the live stream indefinitely; bounded buffering + failure policy
- [ ] Capture stream metadata: route id, stream kind/class, params, terminal size/resize where available, user, connection
- [ ] Add metrics: recording active count, bytes written, failures
- [ ] Audit start/finalize/failure/read separately from route audit

## Definition of done
A fake WS stream can be recorded by the core wrapper with no plugin-specific code, and live streaming behavior remains correct under recorder backpressure/failure tests.

## Tests (gate)
- [ ] Unit: wrapper records upstream output and browser input events with monotonic timestamps
- [ ] Required-recording failure denies the stream before handler execution
- [ ] Optional-recording failure allows stream but audits/marks recording failed per policy
- [ ] Backpressure test proves slow storage cannot deadlock the stream
- [ ] Race test around concurrent stream close and recorder finalize

## Notes
Core recording sees byte streams. Format-specific recorders interpret those bytes only when the stream class supports it.
