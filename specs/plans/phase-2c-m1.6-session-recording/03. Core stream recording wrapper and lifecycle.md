# Phase 2c · Step 3 — Core stream recording wrapper and lifecycle

**Milestone:** M1.6 · **Status:** ✅ Done (2026-05-25)
**Depends on:** 2.8, 2.9, 2c.1, 2c.2 · **Spec refs:** [v2 §8.1](../../v2.md), [v2 §9.5](../../v2.md)

## Goal
Record stream traffic at the core wrapper boundary without making SSH, VNC, or
RDP special cases. The wrapper decides whether recording is active from plugin
capability + connection policy + user permissions.

## Policy semantics (auto = forced)
- `disabled` — never record; no tap installed.
- `auto` — **forced**: for terminal streams, recording is armed when the stream
  opens but the recording row/blob starts only on the first real terminal input
  (resize/control frames do not count). The user cannot stop it. If the recorder
  cannot start on that first interaction, the stream fails closed before the input
  reaches the upstream. Non-terminal forced streams still start up front when the
  stream opens.
- `manual` — a recording tap is installed but idle; the user starts/stops it
  through control endpoints. Stopping finalizes the recording.

## Definition of done

## Notes
Core recording sees byte streams. Format-specific recorders interpret those bytes only when the stream class supports it.
