# Phase 6 · Step 3 — Database safety controls

**Milestone:** M5 · **Status:** ✅ Done — 2026-05-26
**Depends on:** 6.2 · **Spec refs:** [v2 §9.6](../../project.md), [v2 §9.5](../../project.md)

## Goal
The non-negotiable DB guardrails from §9.6.

## Checklist
- [x] Read-only mode toggle (per connection)
- [x] Query timeout + row limit (configurable, enforced server-side)
- [x] Dangerous-statement detection (`UPDATE/DELETE/TRUNCATE/DROP`) → confirm/approval, tagged `destructive`
- [x] Optional approval workflow hook for production connections
- [x] **Every query audited** with statement metadata, rows affected, duration, and result
- [x] Result redaction hooks where configured

## Definition of done
A destructive statement is gated behind confirm/approval in a protected connection; read-only mode blocks writes; all queries are audited.

## Notes
M5 implements the approval hook as a server-side destructive-statement
confirmation gate exposed through the generic `query_editor` protocol. A later
multi-user approval queue can attach at the same per-query audit/decision point
without changing PostgreSQL route handlers.
