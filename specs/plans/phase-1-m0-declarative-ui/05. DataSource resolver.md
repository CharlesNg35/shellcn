# Phase 1 · Step 5 — DataSource resolver

**Milestone:** M0 · **Status:** ☐ Not started
**Depends on:** 1.4 · **Spec refs:** [v2 §6.2](../../v2.md), [v2 §7.1](../../v2.md), [v2 §7.2](../../v2.md), [v2 §7.3](../../v2.md)

## Goal
One client util that turns a `DataSource` (`RouteID` + params) into a request,
interpolating `${resource.*}`, with pagination and watch support. The frontend
never builds URLs by hand.

## Checklist
- [ ] Param interpolation (`${resource.uid}`, `${resource.namespace}`, statics)
- [ ] Resolve `RouteID` → `/api/connections/{id}/x/{routeID}` (mock base in M0), sending route params as **`?p.<name>=<value>`** and list controls as `cursor`/`limit`/`filter`/`sort` (v2 §7.1)
- [ ] WS: request a ticket (`POST …/tickets` with routeID + resolved params), open `wss://…/x/{routeID}?p.<name>=…&ticket=…` (v2 §7.4)
- [ ] Cursor pagination client (`PageRequest` → `Page<T>`), filter + sort
- [ ] Watch subscription helper (WS → `ResourceEvent` stream)
- [ ] Base is swappable mock ↔ real with zero panel changes

## Definition of done
Table and Tree panels load data via the resolver against the mock server; pagination and a watch subscription work; the resolver emits the `p.`-prefixed param scheme.

## Tests (gate)
- [ ] Unit: `RouteID` + params → correct URL (`p.` prefix; reserved list keys not collided)
- [ ] Unit: `${resource.*}` interpolation; missing param errors cleanly (no silent blank)
- [ ] Unit: pagination follows `NextCursor`; watch reconnects

## Notes
Keep interpolation tiny and typed — it is **not** a scripting language (§1 principle 3).
