# Phase 1 · Step 4 — Manifest renderer and panel dispatch

**Milestone:** M0 · **Status:** ✅ Done (2026-05-25)
**Depends on:** 1.3 · **Spec refs:** [v2 §6.2](../../project.md), [v2 §6.3](../../project.md), [v2 §8.4](../../project.md), [v2 §12](../../project.md)

## Goal
`ConnectionWorkspace.vue` reads a projection and renders the right layout, then
dispatches each tab/detail to a panel component by `PanelType`.

## Checklist
- [x] Render `Layout`: `LayoutTabs` → tab bar; `LayoutSidebarTree` → tree + content
- [x] Panel registry: `PanelType` → Vue component, resolved via **`defineAsyncComponent` (lazy/dynamic import)** so heavy panels load only on first use (v2 §12.1)
- [x] Shared `Icon` component rendering `name`/`url`/`base64`/`emoji` with fallback (used by tabs/tree/actions)
- [x] Resource row/tree-node click → `DetailView` with its tabs
- [x] Agent-mode connection with no live tunnel → render `EnrollPanel` instead of tabs/tree
- [x] Unknown `PanelType` → graceful fallback (don't crash); loading/skeleton state while a lazy panel resolves

## Definition of done
SSH fixture shows tabs, Docker fixture shows tree + container detail, Proxmox shows VM detail tabs — all driven purely by the projection, zero per-plugin code; heavy panels are not in the initial bundle.

## Tests (gate)
- [x] Component test: registry maps each `PanelType` to a component; unknown type → fallback
- [x] Component test: `Icon` renders all four types and falls back on empty/failed
- [x] Build check: a panel chunk is fetched only when its tab is opened

## Notes
This component is the heart of "adding a plugin needs zero frontend work" (§12). Keep it data-driven only. See [`../TESTING.md`](../TESTING.md).
